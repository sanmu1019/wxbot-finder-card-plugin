# 实际调用链与结论

## 核心结论

当前机器人解析链路涉及三种不同登录态或服务，不能混为一谈：

| 能力 | 登录态来源 | 是否生成 sph |
|---|---|---|
| 接收视频号卡片 XML | 机器人微信号协议 | 否 |
| Finder 短链网关 | 视频号管理后台 Chromium | 是 |
| protocol-core Finder 详情 | 机器人微信号协议 | 否 |
| wxsph-api 元宝解析 | 元宝网页会话 | 否，只消费 sph |

因此，“机器人协议生成短链后再解析”不是准确描述。

准确描述是：

```text
机器人协议提供卡片 XML
-> Finder 管理后台网关生成 sph
-> 元宝解析 sph
```

## 主链路

### 1. 机器人收到消息

wxbot 将消息内容放入事件上下文。插件会读取：

```text
content
original_content
outer_content
```

引用卡片的原始 XML 通常位于后两个字段之一。

### 2. 提取卡片字段

插件拼接原始内容为 `source_blob`，并从多层 HTML 转义、URL 编码或 CDATA 中
提取：

```text
objectId
objectNonceId
nickname
desc
finderUsername
```

只有 `objectId + objectNonceId` 同时存在，才可以调用短链网关。

### 3. 调用 Finder 短链网关

插件发送：

```text
POST /api/v1/finder/share-url
```

请求体：

```json
{
  "object_id": "OBJECT_ID",
  "object_nonce_id": "OBJECT_NONCE_ID",
  "scene": 40
}
```

Finder 网关内部的 Chromium 已登录视频号管理后台，它调用页面 webpack
service 的 `getObjectShortLink`，返回：

```text
https://weixin.qq.com/sph/SHORT_CODE
```

该步骤使用的是管理后台登录态，不是机器人协议登录态。

### 4. 调用 wxsph-api

插件把 `sph` 发送到：

```text
GET /api/wxsph?url=<sph>
```

当前 `wxsph-api` 实现使用元宝会话解析：

- 标题
- 作者
- 封面
- 视频候选地址

元宝只消费已经存在的 `sph`，不能从卡片 `objectId` 推算短链。

### 5. 下载和发送

插件依次尝试解析结果中的媒体候选：

1. 下载到 wxbot 临时目录。
2. 检查文件头是否为视频。
3. 必要时处理 Finder 加密媒体。
4. 调用 `send_video_msg`。
5. 视频发送失败时降级为文件发送。
6. 最后删除临时文件。

## 可选协议直刷兜底

当以下条件同时成立时：

```text
没有 sph
有 objectId + objectNonceId
finder_protocol_enabled=true
```

插件调用：

```text
POST /Finder/GetCommentDetail
```

protocol-core 使用当前机器人微信号的协议登录态调用 Finder CGI 3906，刷新：

```text
媒体基础 URL
url_token
decode_key
标题和作者
```

这条路径直接得到媒体信息，不经过 `sph` 和元宝。

## 为什么保留两条路径

### Finder 短链路径

优点：

- 得到标准可分享 `sph`。
- 与机器人协议实现解耦。
- 可继续复用现有 `wxsph-api`。

限制：

- 需要单独维护管理后台 Chromium 登录状态。
- 页面前端更新可能影响 webpack service 定位。

### 协议直刷路径

优点：

- 不依赖管理后台浏览器。
- 可以直接刷新同批次媒体 URL 和解密 Key。

限制：

- 依赖特定 protocol-core 私有接口。
- 不会生成可分享 `sph`。
- 更依赖微信协议实现和 protobuf 字段。

## 插件真实优先级

代码按以下顺序处理：

```text
消息已有 sph
-> 直接解析

否则有 objectId + objectNonceId 且 bridge 开启
-> Finder 网关生成 sph
-> 解析 sph

否则协议兜底开启
-> Finder/GetCommentDetail
-> 直接下载和发送

否则卡片有完整 CDN URL
-> 尝试直接下载

否则
-> 返回错误提示
```

桥接成功后不会再调用 protocol-core。

## 部署组合

### 推荐完整模式

```text
wxbot
Finder 短链网关
wxsph-api
```

适合需要生成标准短链并发送视频。

### 纯协议模式

```text
wxbot
支持 Finder/GetCommentDetail 的 protocol-core
```

配置：

```json
{
  "finder_bridge_enabled": false,
  "finder_protocol_enabled": true
}
```

适合只需要机器人直接下载发送，不需要 `sph`。

### 只解析现有 sph

```text
wxbot
wxsph-api
```

用户必须提供已经存在的 `weixin.qq.com/sph/...`。
